## The Setup

There's a version of learning to code that happens entirely inside tutorials, and a different version that only happens once you're the one responsible for a project from idea to something that actually runs somewhere.

## What Tutorials Don't Teach

Tutorials are optimized for a clean, linear path: step 1 leads cleanly to step 2. Real projects rarely work that way. The gap between "I followed the tutorial" and "I can build this from scratch" is almost entirely made of decisions tutorials make *for* you — project structure, error handling, what to do when a dependency breaks.

## The Pattern I Noticed

Across enough small projects, a few recurring lessons kept showing up:

1. **Naming things well matters more than it seems.** A confusing file structure slows down every future change.
2. **The boring parts (config, logging, error messages) are usually what determines whether a project survives contact with real use.**
3. **Shipping something small and finished beats an ambitious project that never leaves "in progress."**

## What Failed

Plenty of projects were abandoned halfway — usually the overly ambitious ones started without a clear, small first milestone.

## What I'd Do Differently

Start every new project by writing down the smallest possible version that's still useful, and ship *that* first.

## Conclusion

The tutorials taught syntax. The projects taught judgment — and judgment is the part that doesn't show up in any single lesson.
