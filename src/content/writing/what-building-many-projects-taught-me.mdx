# What Building Many Small Projects Actually Taught Me

There is a specific kind of existential dread that no coding bootcamp prepares you for. 

It's 2 AM. You've just pushed your 47th commit to a side project. The core feature works, but you just realized you hard-coded an API key into a file you're about to push to a public repo. You rename the file, break the imports, and spend the next hour fixing a cascade of errors.

Tutorials are beautifully curated gardens. They hand you a map, a clear blue sky, and a guide who holds your hand at every turn. Real project building is a jungle—and you're hacking through the vines with a machete made of Stack Overflow tabs.

Through the chaos of building dozens of small applications—some successful, most failures—I've learned hard-won lessons that no tutorial can teach you. This is the honest inventory of those scars.

![A chaotic scene of tangled wires, post-it notes, and screens, versus a clean, orderly room.](https://picsum.photos/seed/tutorial_vs_reality/1200/600)

## The Setup: Syntax vs. Judgment

When you learn to code, you measure progress by syntax. "I know how to write a `for` loop," or "I understand closures." But after a few years, you realize that the bottleneck of a successful project is rarely the syntax.

The difference between a "tutorial-follower" and a "shipper" is **judgment**. Judgment is knowing:
- Which dependency is worth adding to your `package.json` (and which will cause a 3-month upkeep nightmare).
- When to abstract code into a new module, and when to duplicate the line once and move on.
- How to write an error message that doesn't just tell the user *what* went wrong, but gives them a clear path to *fix* it.

Tutorials optimize for a clean, linear path. Step 1 leads to Step 2. Real projects are waterfalls of cascading decisions. The gap between "I followed the tutorial" and "I can build this from scratch" is entirely made of the decisions the tutorial made *for* you.

## Lesson 1: Structure is Not Aesthetics, It's Survival

I used to think file structure was a matter of personal preference. You like putting `utils.py` in the root? Great, I'll put it in `helpers/`. 

Then I worked on a project six months later and realized I couldn't find my own database models. I had scattered them across four different directories because they "felt right" at the time.

**The cost of bad naming scales linearly with time.** 

On one project, I named a function `process_data()`. A month later, I needed to update it. I had no idea if it cleaned the data, transformed it, or sent it to an API. I had to read the entire function body—140 lines of spaghetti—just to understand its side effects.

**The fix:** I now treat naming like a product feature. If a file name or function name isn't explicit enough that I could guess its contents without opening it, I rename it before I commit. It sounds pedantic, but it saves the "where-the-hell-is-that-module" panic every single time.

## Lesson 2: The Boring Parts Are the Golden Parts

If you look at a new project, the exciting part is the "magic"—the AI inference, the slick UI animation, the real-time websocket. 

But here's the hard truth: **The boring parts (config, logging, error handling) determine if the project survives contact with real users.**

I built a microservice that summarized news articles. The summarizer worked great. I deployed it, and within an hour, a user uploaded a corrupted PDF. The service crashed. I didn't have structured logging enabled, so I spent 45 minutes tailing raw terminal output to figure out why. 

**My new rule:** I do not write a single line of production code until I have:
1. A `.env` config loader that is environment-aware.
2. A JSON-based logging handler (so I can query `level:ERROR` in my logs).
3. A global exception handler that returns a sane JSON response, not an HTML stack trace.

This "boring" setup takes 15 minutes. It saves 5 hours of debugging on a bad Tuesday.

## Lesson 3: Velocity Beats Ambition (The 80/20 Trap)

Early in my career, I was an "architect." I wouldn't start a project until I had designed the perfect database schema, the most elegant microservices architecture, and a configurable admin panel that could handle a million users.

The problem? I was building a todo-list app.

I spent three weeks building the infrastructure. I wrote zero lines of application logic. The project died in the "design phase."

**The pattern:** The last 20% of polishing an ambitious project often takes 80% of the time. When you have a massive scope, that 80% becomes a soul-crushing mountain of work that never ends.

**The lesson:** Start with the smallest possible version that is still *useful to someone*.

For a recent project (a habit-tracker), instead of building user-auth and a database from scratch, I started with a single JSON file on disk. I shipped it to a friend within 4 hours. They used it, found a bug, and I fixed it. The JSON file was ugly, but the *feedback loop* was active.

Only when the JSON file became too slow did I migrate to PostgreSQL. I didn't waste a single hour on infrastructure that wasn't yet necessary.

## Case Study: The Project That Failed Hard

I built a "social reading" app. Readers could highlight passages in eBooks and see other readers' highlights in real-time.

It was a beautiful concept. But here's what I built first:
1. A user authentication system with OAuth.
2. A beautifully designed profile page.
3. An admin dashboard to moderate users.
4. *The actual reading/highlighting core feature.*

I launched it. The authentication worked perfectly. The profile pictures loaded quickly. But the core feature—the real-time highlighting—crashed after 5 concurrent users because I had over-engineered the database schema and under-engineered the WebSocket connection handling. 

The app died because I built the *shell* of the product before verifying the *core*. It taught me a brutal but necessary lesson: **If you build the admin panel before the feature, you are procrastinating on the hard part.**

## What I'd Do Differently (The New Approach)

Today, I follow a strict **"One-Page MVP"** rule:

1. **Write down the user journey** on a single sheet of paper. 
2. **Find the "Spine"** of the application. What is the single, non-negotiable action the user must be able to do? For a delivery app, it's "order food." For a content site, it's "read an article."
3. **Build the Spine first.** Everything else (auth, payments, admin UI) is scaffolding. Scaffolding can be added later. If the spine doesn't work, the entire building collapses.

By focusing on velocity, I went from finishing 2 projects a year to finishing 12. Not all of them got users. But every single one of them taught me something new.

## Conclusion: The Scars Are the Resume

Tutorials teach you the grammar of a language. But projects teach you the *dialect* of the real world.

You learn that naming is hard, that logging is a lifesaver, and that shipping something "good enough" on Friday is infinitely better than shipping "perfection" on the next fiscal quarter.

The value of building many small projects isn't just the portfolio of code you collect. It's the scars you earn. It's the psychological resilience to start a new project, knowing full well that you *will* hit a wall—but also knowing you have the experience to climb over it or tunnel under it.

If you are hesitating to start that side project, remember: The judgment you build by failing fast and shipping often is the part that doesn't show up in any single lesson. It is the foundation that the lessons are built upon.

### 📚 Personal Takeaways
- **Project Structure matters.** A messy codebase is a moving target.
- **Velocity > Perfection.** You can't fix a project that doesn't exist.
- **Logging isn't optional.** It's your first line of defense in production.
- **Build the core feature first.** The UI can wait.
